## 학습 목표

이 레슨을 마치면 다음을 할 수 있게 됩니다.

- Project가 무엇이고 언제 만드는 것이 좋은지 설명할 수 있다
- 이름·설명·공개 범위를 지정해 새 Project를 만들 수 있다
- Project 지식 베이스에 문서·파일을 올려 Claude가 참조하게 만들 수 있다
- Claude의 동작을 안내하는 효과적인 *project instructions* 를 작성할 수 있다
- (Claude for Work — Team·Enterprise) 동료와 Project를 공유해 협업 환경을 구성할 수 있다

위 영상은 Claude.ai의 Projects 기능을 처음 접하는 사람을 위한 짧은 안내입니다. 새 프로젝트를 만들고 지식을 채워 가는 흐름을 함께 보여 줍니다.

## 핵심 요점

- **Project는 *self-contained 워크스페이스*입니다.** 자체 메모리·대화 기록·지식 베이스·맞춤 지침을 가진 채, 특정 업무 흐름에 전용 환경을 제공합니다.
- **Project knowledge는 Claude의 이해도를 바로 끌어올립니다.** 관련 문서를 한 번 올려 두면 그 프로젝트의 모든 대화에서 자동으로 참조됩니다 — 매번 같은 파일을 다시 첨부할 필요가 없습니다.
- **Project instructions는 Claude의 동작 자체를 다듬어 줍니다.** 톤·전문 수준·응답 스타일·필수 요건을 한 곳에 적어 두면 그 프로젝트의 모든 대화에 자동 적용됩니다.
- **지식이 커지면 자동으로 확장됩니다.** context window 한도에 가까워지면 *Retrieval Augmented Generation (RAG)* 모드가 자동으로 켜져 용량을 최대 10배까지 늘리면서도 응답 품질을 유지합니다.
- **Claude for Work에서는 협업 도구가 됩니다.** 같은 맥락·지침·축적된 지식을 팀이 공유한 채로 일할 수 있습니다.

## Project란?

Claude가 반복적으로 참조해야 할 *지식*을 모아 두고, 특정 주제·업무 영역에 대한 대화들을 한 자리에 정리하며, 같은 컨텍스트 위에서 팀과 함께 일하기 좋은 단위가 Project입니다.

### 언제 만드는 게 좋은가

일회성 질문보다 *지속되는 작업*에 잘 맞습니다. 다음 중 하나라도 해당된다면 Project로 묶어 두는 편이 유리합니다.

- **반복적으로 참고할 자료**가 있는 경우 (회의록, 설문 결과, 리포트, 과거 데이터 등)
- **응답에 일관된 요구**가 있는 경우 (격식체 고정·출처 인용 필수·자체 템플릿 준수 등)
- **팀이 같은 토대 위에서 일해야** 하는 경우

## 첫 Project 만들기

설정 자체는 몇 분이면 끝납니다.

### Step 1 — 프로젝트 만들기

1. 좌측 사이드바의 *Projects* 로 이동하거나 `claude.ai/projects`를 직접 연다.
2. 우측 상단 **+ New Project** 클릭.
3. 의미가 드러나는 이름을 붙인다. *"Q4 마케팅 캠페인"*, *"제품 문서"* 처럼 구체적이면 좋다.
4. 짧은 설명을 적는다. *Claude는 이 설명을 직접 읽지 않지만*, 본인·팀원이 프로젝트 목적을 빠르게 파악하는 데 쓰인다.
5. 공개 범위를 지정한다 — 비공개로 둘지, (Claude for Work) 조직 내에 공유할지 선택한다.

### Step 2 — Project instructions 작성

instructions는 그 프로젝트의 *모든 대화*에 적용되는 동작 지침입니다. **Instructions** 패널을 열어 다음 내용을 채워 두면 좋습니다.

- **무엇을 다루는 프로젝트인가** — 예: *"이 프로젝트는 B2B SaaS 제품의 마케팅 콘텐츠 작성용입니다."*
- **프로세스 지침** — 예: *"먼저 이 청중에게 어울리는 블로그 구조를 잡고, 그다음 본문 초안을 쓴다."*
- **톤·스타일 선호** — 예: *"전문적이지만 대화체. 가능하면 전문 용어는 피한다."*
- **필수 요건** — 예: *"마케팅 카피 끝에는 항상 행동 유도 문구(CTA)를 둔다."*

작성이 끝나면 **Save instructions**. 사용자 *Preferences* 와 *Styles* 와도 함께 작동하므로, 충돌 없이 자연스럽게 누적됩니다.

> Instructions는 *동작 자동화*에도 쓸 수 있습니다 — *"회의록을 업로드하면 다음 템플릿대로 구조화된 요약을 만들어 줘"* 처럼 적어 두면 같은 흐름이 매번 일관되게 돌아갑니다. *Claude의 동작을 프로그래밍한다* 는 감각으로 작성하세요.

### Step 3 — 지식 베이스 채우기

오른쪽 패널의 파일 메뉴에서 **+** 버튼을 눌러 자료를 추가합니다. PDF·DOCX·CSV·TXT·HTML 등 다양한 형식이 들어가고, Google Drive를 연결해 문서를 직접 링크할 수도 있습니다.

올리기 좋은 자료는 다음과 같습니다.

- **참고 자료** — 브랜드 가이드라인, 스타일 가이드, 템플릿
- **배경 자료** — 리서치 보고서, 회의록, 요구사항 문서
- **모범 예시** — Claude가 따라가 주었으면 하는 글의 본보기
- **기술 문서·스펙**

> **Pro tip — 파일 이름은 *서술적*으로.** Claude는 파일 이름을 단서로 적합한 자료를 찾아냅니다. *"Q4-2024-Brand-Guidelines.pdf"* 가 *"document1.pdf"* 보다 훨씬 도움이 됩니다.

### 지식이 커지면 — RAG 자동 전환

자료를 많이 올렸을 때를 걱정할 필요는 없습니다. 지식 베이스가 context window 한도에 가까워지면 Claude가 *RAG (Retrieval Augmented Generation)* 모드로 자동 전환합니다 — 모든 자료를 한 번에 메모리에 올리는 대신 *질문에 가장 관련 있는 부분만* 검색해 응답에 활용합니다. 덕분에 용량은 **최대 10배** 늘어나고, 사용 경험은 평소와 다르지 않습니다. RAG가 켜져 있으면 화면에 표시기가 함께 보입니다.

## 프로젝트 안에서 일하기

프로젝트가 준비되면 그냥 평소처럼 대화를 시작하면 됩니다. 모든 채팅이 자동으로 지식 베이스를 참조하고 instructions를 따릅니다 — 매번 맥락을 다시 설명하지 않아도 됩니다.

## 협업 — Claude for Work

Team·Enterprise 플랜에서는 Project가 *공유 가능한 작업 공간*으로 확장됩니다.

### 권한 레벨 세 가지

1. **Can view.** 콘텐츠·지식·대화를 *볼 수 있고 함께 채팅* 도 가능하지만, 변경은 할 수 없습니다 — 토론권 있는 읽기 전용입니다.
2. **Can edit.** instructions·지식·멤버까지 편집할 수 있는 *전체 협업 권한*. 프로젝트에 능동적으로 기여하는 멤버에게 적합합니다.
3. **Owner.** 프로젝트의 모든 것을 통제합니다. 누가 볼 수 있는지를 결정하고, 특정 인원이나 조직 전체에 공개할 수 있습니다.

### 프로젝트 공유 흐름

- 공유할 프로젝트를 연다.
- 프로젝트 이름 옆 **Share project** 클릭.
- 이름·이메일로 개별 멤버를 추가하거나 이메일 목록을 그대로 붙여넣는다 (대량 공유 시 멤버 화면에는 *Shared with you* 섹션에 자동 노출됨).
- 또는 *"Everyone at [조직 이름]"* 으로 설정해 조직 전체가 Team 탭에서 발견할 수 있게 한다.

공유받은 멤버에게는 알림 메일이 가고, 자기 화면의 *Shared with me* 탭에서 프로젝트를 다시 찾을 수 있습니다.

## 영감용 — 프로젝트 예시

처음 시작할 때 어디부터 손대야 할지 막연하다면 다음 패턴을 참고합니다.

- **Q4 제품 출시.** 제품 스펙·경쟁사 분석·메시징 브레인스토밍 노트를 모아 두면, 이후 어떤 질의·초안 요청에도 일관된 맥락이 깔린 응답이 나옵니다.
- **리서치 허브.** 경쟁사 검토·사용자 인터뷰·고객 피드백을 한 자리에 두고, 출처를 가로지르는 정리·리포트 초안·일관된 권고안을 받아 봅니다.
- **클라이언트 계정 허브.** 클라이언트의 브랜드 가이드라인·과거 산출물·커뮤니케이션 기록을 모아 두고, 제안서·리포트가 항상 그 톤·맥락을 따라가도록 instructions를 적어 둡니다.
- **이벤트 운영.** 장소 계약서·연사 약력·참가자 데이터를 올려 두면, 진행표·참가자 안내·사후 리포트가 같은 테마 안에서 일관되게 만들어집니다.
- **채용 공고 생성기.** 과거 공고·팀 헌장·헤드카운트 요청 문서를 모아, 실제 팀의 일과 문화를 반영한 채용 공고 초안을 함께 다듬어 갑니다.

## 운영 best practice

- **focused 시작 → 점진적 확장.** 한 번에 모든 걸 담는 대형 프로젝트보다, 한 use case에서 시작해 필요할 때 자료를 더해 가는 편이 유지·관리가 훨씬 수월합니다.
- **지식 베이스를 최신으로.** 오래된 자료는 오래된 답을 만듭니다. 주기적으로 검토하고 갱신합니다.
- **명확하고 구체적인 instructions.** 모호한 지시는 결과의 일관성을 깎아 먹습니다.
- **파일 이름은 서술적으로, 관련 파일은 함께.** Claude는 파일 이름과 *근접성*을 단서로 문서 간 관계를 파악합니다.
- **문서 이름으로 직접 가리키기.** 질문할 때 *"Q3 리포트 기준으로, 가장 큰 고객 우려는 무엇이었나요?"* 처럼 구체 파일을 호명하면 검색이 빨라집니다.

## 학습 후 돌아보기

- 지속되는 작업 중 *전용 프로젝트*가 있으면 가장 도움이 될 것은 무엇인지.
- Claude에 매번 다시 올리거나 다시 설명하고 있는 자료가 있다면 어떤 것인지.
- 팀에서 일한다면 *공유 지식·공유 지침* 으로 묶어 두면 좋을 프로젝트가 있는지.
